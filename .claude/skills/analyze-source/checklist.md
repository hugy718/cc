# 分析质量检查清单

分析完成后，逐项检查：

## 完整性

- [ ] 所有相关源文件都已识别并列在索引中
- [ ] 入口点（main/script/CLI）已明确
- [ ] 配置文件已识别并列出关键配置项
- [ ] 数据流从输入到输出的每一步都有覆盖，无跳跃
- [ ] 错误处理/降级路径已记录

## 准确性

- [ ] 每个代码行为描述都有 `文件:行号` 引用
- [ ] 配置项引用包含 YAML key path 和默认值
- [ ] 示例数据前后一致（Step N 输出 = Step N+1 输入）
- [ ] 没有基于猜测的描述——全部来自实际读取的代码

## 可读性

- [ ] 有架构图或流程图（ASCII 或描述）
- [ ] 字段/特征使用表格展示，不是纯文字罗列
- [ ] 每个 Step 都有输入/输出示例
- [ ] 文档按数据流组织，不是按文件组织
- [ ] 中文撰写，代码/配置保持原文

## 复杂工作流（≥4 个类 或 跨 ≥3 个 OO 抽象层）

- [ ] Phase 1 输出了类清单（每个类：file:line / 角色 / 被谁调用 / 调用/拥有谁）
- [ ] 有 Mermaid `classDiagram`：包含关键属性+方法，关系用 `-->` / `o--` / `..>` 区分
- [ ] 有 Mermaid `sequenceDiagram`：至少一条完整成功路径 + 重试/失败分支（`alt/opt`）
- [ ] Note over 标注每一跳写入到下游（训练样本/状态）的字段
- [ ] Mermaid 代码块内 **无分号** `;`（分号是 Mermaid 语句分隔符，会拆断 Note/消息）
- [ ] participant 别名 **无** `<br/>` / 括号 / 空格（多行说明用 `Note over` 另起）
- [ ] 消息/注释内 **无** Unicode 箭头 `→`、中点 `·`、花/方括号 `{}` `[]`
- [ ] 单分支条件用 `opt` 而非 `alt ... end`（无 `else`）

## 敏感信息脱敏

- [ ] 无 API keys / tokens / secrets 的实际值（长度 > 16 的 hash 类字符串）
- [ ] 无 `bytedance.net` / `byteintl.net` / `tiktok-row.net` 等内部完整域名
- [ ] 无 Redis PSM 名称或连接字符串
- [ ] 无硬编码的用户名 / 服务账号名
- [ ] 无 `app_id` / `app_key` / `app_secret` 的实际值
- [ ] 无 JWT token 或 bearer token 值
- [ ] 无内部 IP 地址或基础设施端口号
- [ ] 配置示例使用 `<REDACTED>` 占位符，不是真实值
- [ ] 代码片段使用变量引用（如 `self.secret`），不是字面量

## 实用性

- [ ] 新工程师读完能理解该子系统如何工作
- [ ] 能找到修改任何配置/行为所需的文件和行号
- [ ] 常见调试场景（日志位置、错误排查）有提及
